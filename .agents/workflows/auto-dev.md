# 自动开发工作流 (Auto-Dev Pipeline)

当用户输入 `/auto-dev [TODO内容]` 时，严格按顺序执行以下阶段。
当用户输入 `/auto-dev resume` 时，跳转到【会话恢复协议】。

> ⚠️ **前置要求**：`/auto-dev` 的输入必须是经过 `requirement-clarification` 工作流确认的《需求规格确认书》，或明确的单点 Bug Fix。直接输入模糊需求描述时，必须先触发 `requirement-clarification`，不得跳过。

---

## 会话恢复协议 (Session Resume)

> 用于会话中断后从断点继续，而非从头开始。

1. 通过工具读取 `tmp/.agent-session.md`
2. 若文件不存在：告知用户"未找到存档，请重新启动 `/auto-dev [TODO内容]`"，终止
3. 若文件存在：
   - 读取"当前阶段"、"规范快照摘要"、"问题全量清单"、"已完成轮次"
   - 输出恢复确认：`[RESUMED] 从第 N 轮继续，剩余 P0: x 个，P1: x 个`
   - 直接进入对应阶段的断点位置继续执行
   - **不重新执行已完成的阶段**

---

## 阶段零：规范预加载 (Pre-read)

> 所有阶段开始前必须执行，不可跳过。

1. **读取项目约定文档**（每次启动必读，不可跳过）：
   - 读取 `docs/conventions.md` 的 `## 核心约定速查` 区块（止于第一条分隔线 `---`），将其中约定加入本次《规范快照》
   - **NEVER 读取分隔线之后的完整内容**（Verify 阶段发现潜在违规时按需读取对应章节）
   - 若文件不存在 → 跳过，不报错

2. **按需路由加载 SKILL.md**（只读命中项，未命中跳过）：

   | 任务涉及 | 加载目标 |
   |---------|---------|
   | 前端 | `.agents/rules/frontend-ui.md` + `generate-test-skeleton/SKILL.md` + `frontend-ux-evaluator/SKILL.md` |
   | 数据库 | `.agents/rules/db.md`（路由器）+ `export-db-indexes/SKILL.md` |
   | 权限 | `project.config.json` 中 `auth.docs_path` 指向的文档 |
   | 任何任务 | `commit-with-affects/SKILL.md` |

   **NEVER 加载未在上表命中的 SKILL.md**，需要时按需读取，不在此阶段预加载。

3. **决策记录预读**：
   - 读取 `docs/decisions/README.md` 索引表
   - 对照本次任务涉及的模块路径，找出命中 `affects` 字段的决策文件
   - **只读每个命中文件的 `<!-- QUICK: -->` 行**（frontmatter 之后第一行），提取 NEVER 条目加入快照禁止清单
   - 需要完整 ADR 上下文时（如大幅重构相关模块）才读全文
   - 若索引表为空或不存在 → 跳过，不报错

3. **生成《本次任务规范快照》**，格式约束如下：
   - 本次涉及的技术层（前端 / 后端 / DB / 权限）
   - 每层强制约束：**最多 5 条**，每条不超过 20 字，只写本任务特有的
   - 禁止事项清单：只列本任务相关的，通用红线不重复（已在规则文件）
   - token / 设计系统定义位置：**只记文件路径**，不复制内容
   - **快照总长度控制在 30 行以内**

4. 快照生成后即封存。**NEVER 在阶段一至五的任何步骤中重新读取规则文件或 SKILL.md**。所有规范引用必须来自本快照。唯一例外：Verify 阶段需核查源文件具体实现时，可只读目标文件，不更新快照。

## 阶段一：立项与基线保底 (Pre-flight)

1. **触发 `/dev-flow` 建立开发沙盒**：
   - 优先通过 IDE 功能或 AST 进行静态影响分析，禁止盲目依赖 grep
   - 切换到新的 `feature/` 分支

2. **生成《问题全量清单》**：
   - 扫描当前代码库与本次 TODO 相关的已有问题
   - 按 P0（阻塞）/ P1（影响质量）/ P2（优化）分级
   - 标注问题间依赖关系（上游问题先修）
   - 此清单贯穿全流程，新发现问题只能入队不能插队

---

## 阶段二：方案设计与人工卡点 (Plan & Review)

> ⚠️ 核心原则：严禁"闭眼直接写代码"。

1. **出具方案**，方案必须包含：
   - 技术修改方案
   - 改动影响范围评估（涉及哪些文件/模块）
   - 与《规范快照》的冲突检查（有冲突必须在方案里说明如何处理）
   - 本次不处理的 P2 问题清单（透明告知）
   - **⬡ 组件复用检查（涉及前端时必填）**：
     ```
     - 新建：ComponentName — 已查无同类，理由：xxx
     - 复用：ExistingComponent — 扩展 props: xxx
     ```
   - **⬡ 决策草稿（条件附加）**：若本次方案中存在以下任一情况，在方案末尾追加决策草稿块：
     - 选择了有明显替代方案的技术路线
     - 故意绕开通用模式（如不走统一错误处理中间件）
     - 某模块引入了"不能随意改动"的约束
     - 依赖了一个将来可能消失的外部限制

     草稿格式：
     ```
     ⬡ 待记录决策：[topic]
     决定：一句话描述
     原因：关键驱动因素
     禁止：NEVER ... 因为 ...
     ```

2. **强制挂起审核 (Blocked on User)**：
   方案输出后必须停止，等待用户明确回复"同意" / "LGTM" / "可以开始"后才进入阶段三。

   若方案中包含决策草稿，用户确认方案的同时视为确认决策草稿，**立即调用 `record-decision` skill 写入 `docs/decisions/`**，无需再次询问。

## 阶段三：带监控循环编码 (Execute) — Ralph-loop 模式

> 阶段三整体采用自循环模式，每轮 Assess → Act → Verify → Log。

### 全栈功能执行顺序（Frontend-First 策略）

当《需求规格确认书》中技术层同时勾选了【前端】和【后端】时，**强制按以下顺序分两轮执行**，不得并行：

```
第一轮：前端开发
  → 触发 frontend-tdd 工作流（按组件节拍：写测试→实现→UX评估→人工卡点）
  → 所有前端组件通过 UX 评估并获得人工确认

第二轮：后端开发
  → 从已确认的前端交互中反推 API 契约（字段、状态码、错误格式）
  → 再进入 Ralph-loop 实现后端逻辑

```

> **为什么前端先行**：后端 API schema 应从已验证的 UI 交互中反推，而非凭假设设计。前端确认后，API 契约才是真实需求，避免后端二次返工。

### 每轮循环结构：

**Assess** — 确定本轮范围：
- 从《问题全量清单》取出当前批次（P0 优先，再 P1）
- 明确本轮涉及的文件和不涉及的文件
- 对照《规范快照》确认本轮改动方向

**Act** — 执行改动，自动挂载对应防线：

- **如果修改了数据库模型**：
  - 强制应用 `db.md` 规则，确保字段合法
  - 立刻执行 `export-db-indexes`，提取 SQL 到分片部署脚本

- **如果修改了前端源码**：
  - 强制对照《规范快照》中的 frontend-ui 约束
  - **仅当 `css_framework == "tailwind"`**：禁止物理色（`bg-blue-500` 等），只用语义变量；新增 token 必须同步写入 `tailwind.config.js`
  - 所有前端框架均适用：禁止裸 hex / rgb 硬编码颜色值

- **如果有权限开发要求**：
  - 参照项目权限实现文档执行

**Verify** — 三层验证：

  层 1 — 改动本身：
  - 是否解决了本轮目标问题
  - 是否符合《规范快照》所有约束
  - 新增内容命名是否遵循项目约定

  层 2 — 影响范围：
  - 变更是否超出本轮 Scope
  - 意外影响的文件是否需要额外处理

  层 3 — 新问题检测：
  - 发现的新问题加入《问题全量清单》并标注轮次来源
  - 不在本轮处理，按优先级排队

**Log** — 结构化记录 + **持久化存档**：
- 本轮解决的问题（来自全量清单）
- 新发现并入队的问题
- 全量清单剩余进度（P0: x/x, P1: x/x）
- 下一轮任务预告
- **经验总结触发检测**：若本轮出现以下任一情况，必须在 Log 末尾附上 `[KNOWLEDGE_UPDATE]` 建议文本：
  - 修复了非显而易见的 Bug
  - AI 的假设被验证错误（踩坑）
  - 某个规范在本项目中需要例外处理
  - 测试发现了隐患（不是功能错误，而是潜在风险）

- **决策草稿触发检测**：若本轮实现中发现隐性约束（如绕开通用模式、第三方限制），且阶段二未覆盖，在 Log 末尾附上 `⬡ 待记录决策` 草稿块，留至阶段五兜底处理。

- **约定提议触发检测**：若本轮发现项目中存在重复模式但 `docs/conventions.md` 未覆盖（如"所有弹窗都用某组件但没有写成约定"），在 Log 末尾附上：
  ```
  [CONVENTION_PROPOSAL] [模块] 建议约定：xxx — 来源：本轮开发观察
  ```
  由人工决定是否追加到 `docs/conventions.md` 增量追加区。

> 🔒 **每轮 Log 结束后，必须立即将以下内容写入 `tmp/.agent-session.md`（覆盖写）：**
>
> ```markdown
> # Agent Session State
> updated: {ISO timestamp}
> phase: 3-executing
> round: {N}
>
> ## 规范快照摘要
> tech_stack: {前端/后端/DB}
> 禁止项: {最多3条，每条≤20字}
> token定义: {文件路径}
>
> ## 问题全量清单
> ### P0（未解决）
> - [ ] {问题描述}
> ### P0（已解决）
> - [x] {问题描述}
> ### P1（未解决）
> - [ ] {问题描述}
> ### P2（暂缓）
> - [ ] {问题描述}
>
> ## 执行记录
> 累计：已解决 P0×{n} P1×{n}，共 {N} 轮
> - Round {N-2}: {一句话摘要}
> - Round {N-1}: {一句话摘要}
> - Round {N}:   {一句话摘要}（当前轮）
>
> ## 下一轮任务
> {具体描述}
> ```
>
> **执行记录只保留最近 3 轮 + 累计行**，更早轮次滚出不保留。
> 此文件是唯一的会话存档，上下文压缩或会话中断后均可通过 `/auto-dev resume` 从此恢复。

### 循环退出条件：
- 《问题全量清单》P0 全部解决
- 本轮无新增 P0 问题
- 所有改动通过三层 Verify

## 阶段四：人工核验 (Manual Verification)

1. 向用户输出结构化核验报告：

   ```
   ## 本次改动核验报告

   ### 改动资产与路由总览
   - **前端交互入口**：[新增的页面路由、UI 组件入口或点击交互点]
   - **后端 API 接口**：[新增或修改的 RESTful 调用路径及 Method]
   - **数据与存储层**：[表结构、索引或缓存变动]
   - **底层文件变动**：[文件路径] — [简短技术改动摘要]

   ### 规范符合性
   - [ ] 无 Tailwind 物理色
   - [ ] 无裸 hex / rgb
   - [ ] 所有新增 token 已在配置文件定义
   - [ ] 数据库字段符合 db.md 规范
   - [ ] 权限逻辑符合项目权限约定

   ### 遗留问题（P2，本次未处理）
   - [问题描述] — [建议处理时机]

   ### 建议人工验证的交互场景
   - [场景描述]
   ```

2. 由人工决定随后的运行与界面预览验收。

---

## 阶段五：合规收口入库 (Commit)

1. **强制复习规范**：
   - 重新读 `commit-with-affects/SKILL.md` 原文（不跳过）
   - 对照《规范快照》做最终检查

2. 按规定组装 commit：
   - 正确填入影响面评估和优先级标度
   - commit message 包含本次解决的问题清单摘要

3. 推入 Feature 分支完成入库。

4. 输出《本次开发完整报告》：
   - 执行了几轮 Execute 循环
   - 最终交付的业务能力汇总
   - 解决的问题清单（P0/P1）
   - 遗留的 P2 清单
   - 对《规范快照》的补充建议（发现但原规范未覆盖的内容）
   - 建议沉淀为新 SKILL.md 的最佳实践

5. **决策兜底检测**（条件触发，捕漏用）：
   检查实现过程中是否出现阶段二未能预见的决策点：
   - 实现中才暴露的隐性约束（如第三方 SDK 限制、并发边界）
   - 用户在本次任务中说了"这里要注意"、"不要改这里"之类的话
   - 某个假设被推翻后，选择了与原方案不同的路线

   若存在且**阶段二未已记录**，输出提议草稿并等待人工确认后写入。
   若阶段二已记录所有决策 → 跳过，不重复提示。

6. 删除 `tmp/.agent-session.md`（任务已完成，存档不再需要）。

7. 退出自动控制流，交接用户触发 `/production-release`。
