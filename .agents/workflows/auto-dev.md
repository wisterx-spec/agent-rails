# 自动开发工作流 (Auto-Dev Pipeline)

当用户输入 `/auto-dev [TODO内容]` 时，严格按顺序执行以下阶段。

---

## 阶段零：规范预加载 (Pre-read)

> 所有阶段开始前必须执行，不可跳过。

1. **强制读取所有相关 SKILL.md**（每次启动都重新读，不依赖记忆）：
   - `.agents/skills/` 下与本次任务相关的所有 SKILL.md
   - `commit-with-affects/SKILL.md`
   - 如涉及前端 → `frontend-ui.md` 规则
   - 如涉及权限 → 项目权限实现文档
   - 如涉及数据库 → `db.md` 规则

2. **生成《本次任务规范快照》**，包括：
   - 本次涉及的技术层（前端 / 后端 / DB / 权限）
   - 每层对应的强制约束（从 SKILL.md 提取关键规则）
   - 禁止事项清单
   - 项目特定 token 定义位置和现有约束

3. 输出《规范快照》供后续所有阶段引用，后续每个阶段 Act 前必须对照此快照，不重新读但必须引用。

---

## 阶段一：立项与基线保底 (Pre-flight)

1. **触发 `/dev-flow` 建立开发沙盒**：
   - 优先通过 IDE 功能或 AST 进行静态影响分析，禁止盲目依赖 grep
   - 切换到新的 `feature/` 分支

2. **生成《问题全量清单》**：
   - 扫描当前代码库与本次 TODO 相关的已有问题
   - 按 P0（阻塞）/ P1（影响质量）/ P2（优化）分级
   - 标注问题间依赖关系（上游问题先修）
   - 此清单贯穿全流程，新发现问题只能入队不能插队

---

## 阶段二：方案设计与人工卡点 (Plan & Review)

> ⚠️ 核心原则：严禁"闭眼直接写代码"。

1. **出具方案**，方案必须包含：
   - 技术修改方案
   - 改动影响范围评估（涉及哪些文件/模块）
   - 与《规范快照》的冲突检查（有冲突必须在方案里说明如何处理）
   - 本次不处理的 P2 问题清单（透明告知）

2. **强制挂起审核 (Blocked on User)**：
   方案输出后必须停止，等待用户明确回复"同意" / "LGTM" / "可以开始"后才进入阶段三。

---

## 阶段三：带监控循环编码 (Execute) — Ralph-loop 模式

> 阶段三整体采用自循环模式，每轮 Assess → Act → Verify → Log。

### 每轮循环结构：

**Assess** — 确定本轮范围：
- 从《问题全量清单》取出当前批次（P0 优先，再 P1）
- 明确本轮涉及的文件和不涉及的文件
- 对照《规范快照》确认本轮改动方向

**Act** — 执行改动，自动挂载对应防线：

- **如果修改了数据库模型**：
  - 强制应用 `db.md` 规则，确保字段合法
  - 立刻执行 `export-db-indexes`，提取 SQL 到分片部署脚本

- **如果修改了前端源码**：
  - 强制对照《规范快照》中的 frontend-ui 约束
  - 禁止 Tailwind 物理色，只用语义变量
  - 禁止裸 hex / rgb / 无效 token
  - 新增 token 必须同步写入配置文件

- **如果有权限开发要求**：
  - 参照项目权限实现文档执行

**Verify** — 三层验证：

  层 1 — 改动本身：
  - 是否解决了本轮目标问题
  - 是否符合《规范快照》所有约束
  - 新增内容命名是否遵循项目约定

  层 2 — 影响范围：
  - 变更是否超出本轮 Scope
  - 意外影响的文件是否需要额外处理

  层 3 — 新问题检测：
  - 发现的新问题加入《问题全量清单》并标注轮次来源
  - 不在本轮处理，按优先级排队

**Log** — 结构化记录：
- 本轮解决的问题（来自全量清单）
- 新发现并入队的问题
- 全量清单剩余进度（P0: x/x, P1: x/x）
- 下一轮任务预告

### 循环退出条件：
- 《问题全量清单》P0 全部解决
- 本轮无新增 P0 问题
- 所有改动通过三层 Verify

---

## 阶段四：人工核验 (Manual Verification)

1. 向用户输出结构化核验报告：

   ```
   ## 本次改动核验报告

   ### 改动资产与路由总览
   - **前端交互入口**：[新增的页面路由、UI 组件入口或点击交互点]
   - **后端 API 接口**：[新增或修改的 RESTful 调用路径及 Method]
   - **数据与存储层**：[表结构、索引或缓存变动]
   - **底层文件变动**：[文件路径] — [简短技术改动摘要]

   ### 规范符合性
   - [ ] 无 Tailwind 物理色
   - [ ] 无裸 hex / rgb
   - [ ] 所有新增 token 已在配置文件定义
   - [ ] 数据库字段符合 db.md 规范
   - [ ] 权限逻辑符合项目权限约定

   ### 遗留问题（P2，本次未处理）
   - [问题描述] — [建议处理时机]

   ### 建议人工验证的交互场景
   - [场景描述]
   ```

2. 由人工决定随后的运行与界面预览验收。

---

## 阶段五：合规收口入库 (Commit)

1. **强制复习规范**：
   - 重新读 `commit-with-affects/SKILL.md` 原文（不跳过）
   - 对照《规范快照》做最终检查

2. 按规定组装 commit：
   - 正确填入影响面评估和优先级标度
   - commit message 包含本次解决的问题清单摘要

3. 推入 Feature 分支完成入库。

4. 输出《本次开发完整报告》：
   - 执行了几轮 Execute 循环
   - 最终交付的业务能力汇总
   - 解决的问题清单（P0/P1）
   - 遗留的 P2 清单
   - 对《规范快照》的补充建议（发现但原规范未覆盖的内容）
   - 建议沉淀为新 SKILL.md 的最佳实践

5. 退出自动控制流，交接用户触发 `/production-release`。
