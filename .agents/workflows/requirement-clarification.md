# Requirement Clarification 需求澄清工作流

> **适用时机**：收到任何开发需求时，先执行本协议，输出《需求规格确认书》后才允许进入 `/auto-dev` 或 `/dev-flow`。
>
> **可跳过条件**：需求来自已有的《需求规格确认书》，或任务是单文件 Bug Fix（明确的报错 + 明确的期望行为）。

---

## 阶段一：歧义扫描 (Ambiguity Scan)

收到需求后，AI **不得立即开始设计方案**，必须先扫描以下维度，列出所有需要澄清的问题：

### 1.1 范围边界
- 功能的起始点和终止点是什么？（用户从哪里触发，结果在哪里呈现？）
- 哪些相关功能**不在**本次范围内？（明确排除项防止范围蔓延）
- 影响哪些已有功能？（会不会改变现有用户的操作流程？）

### 1.2 验收标准
- 怎么算"完成"？（具体的、可验证的标准）
- 异常路径的期望行为是什么？（失败时显示什么？数据不存在时怎么处理？）
- 性能要求？（响应时间、并发量、数据规模上限）

### 1.3 技术约束
- 需要兼容哪些已有系统/接口？
- 是否有不可修改的外部依赖（第三方 API、固定数据格式等）？
- 有无权限隔离要求？（哪些角色可以访问？）

### 1.4 优先级与阶段划分
- 必须在本次交付的 MVP 功能是哪些？
- 哪些是"有更好，没有也行"的增强项？
- 是否需要分多次迭代交付？

---

## 阶段二：结构化提问 (Structured Q&A)

将阶段一扫描到的歧义点整理为**有编号的问题清单**，输出给用户：

```
## 需求澄清问题

本次需求：{{用户原始需求描述}}

在开始开发前，我需要确认以下问题：

**范围**
Q1. [问题]
Q2. [问题]

**验收标准**
Q3. [问题]

**技术约束**
Q4. [问题]

请逐一回答，或告知哪些问题可以由我自行判断。
```

**提问原则**：
- 每次提问不超过 **6 个核心问题**（超出的非关键问题 AI 自行做合理假设并在规格书中注明）
- 问题必须是**影响实现路径**的真实歧义，不提可以靠看代码自行判断的问题
- 如果某个维度显而易见，不提，直接在规格书中写"基于现有项目约定"

---

## 阶段三：需求规格书生成 (Spec Generation)

用户回答后，AI 生成《需求规格确认书》：

```markdown
# 需求规格确认书

需求名称：{{功能名}}
版本：v1.0
日期：{{today}}

## 功能描述
{{1-3 句话描述核心功能，用用户语言而非技术语言}}

## 交付范围（In Scope）
- {{具体功能点 1}}
- {{具体功能点 2}}

## 明确排除（Out of Scope）
- {{排除项 1}} — 理由：{{为什么不在本次}}
- {{排除项 2}}

## 验收标准
1. {{可验证的场景描述，如：用户点击X按钮后，Y秒内看到Z结果}}
2. {{异常场景：当数据为空时，显示EmptyState组件而非空白}}
3. {{权限场景：仅 admin 角色可见该入口}}

## 技术层涉及
- [ ] 前端（页面/组件）
- [ ] 后端（API/Service）
- [ ] 数据库（表结构变更）
- [ ] 权限（角色控制）

## AI 自行假设（未问用户的判断）
- {{假设内容}} — 依据：{{现有约定/常识/项目规范}}

## 开发顺序
{{如果涉及前端+后端：前端 UI → AI 自审 → 你确认 → 后端}}
{{如果仅后端：直接进入 auto-dev}}

## 遗留待定（Parking Lot）
- {{暂时搁置的需求点，留待后续迭代}}
```

---

## 阶段四：人工确认卡点 (Human Sign-off)

**强制挂起**，等待用户明确回复：

- 用户回复"确认" / "OK" / "可以开始" → 进入 `/auto-dev [需求规格确认书内容]`
- 用户有修改 → 更新规格书，重新确认
- 用户要求直接开始 → 注明"用户选择跳过规格确认，自行承担需求理解偏差风险"

> 确认后的规格书作为整个开发周期的"合同"——后续所有阶段的验收标准以此为准，不得随意扩展范围。

---

## 约束

- **NEVER** 在规格书确认前开始任何实现（包括"先看看代码"之外的写入操作）
- **NEVER** 在提问阶段超过 6 个问题（超过说明需求太模糊，应先让用户精炼描述）
- **MUST** 规格书中的验收标准必须可验证（不允许"用户体验良好"这类无法量化的描述）
- **MUST** 每次变更需求范围时，更新规格书版本号（v1.1, v1.2...）并记录变更原因
