# 方案评审卡点 (Proposal Gate)

> 本文件被 auto-dev Phase 2 和 dev-flow Step 3 引用。
> 核心原则：严禁"闭眼直接写代码"。
> 红线：本文档未经人工确认，不允许执行 impact-analysis，不允许生成测试骨架，不允许写任何业务代码。

---

## 跳过条件

以下场景跳过方案评审，直接出具简化技术方案进入人工确认：
- Bug fix 且预计改动文件 ≤ 2 个
- 纯样式 / 纯文案调整

不满足跳过条件时，必须执行完整方案评审。

---

## Step 1：读取上下文

按顺序读取以下内容，构建完整的决策背景：

1. `project.config.json` — 获取技术栈、目录结构
2. `docs/decisions/README.md` — 获取历史架构决策，**本方案不得与已有决策冲突，冲突必须在 Step 2 中显式说明**
3. 需求规格说明书（来自 requirement-clarification 输出）

---

## Step 2：生成方案评审文档

调用 `proposal-review` skill（`/proposal-review`），生成方案评审文档。

> auto-dev 在标准路径下，需在方案评审文档末尾**追加**以下技术补充项。

### auto-dev 技术补充（仅 auto-dev 执行）

在 proposal-review 生成的文档末尾追加：

**规范冲突检查**：与《规范快照》的冲突检查（有冲突必须在方案里说明如何处理）

**P2 透明告知**：本次不处理的 P2 问题清单

**⬡ 组件复用检查（涉及前端时必填）**：
```
- 新建：ComponentName — 已查无同类，理由：xxx
- 复用：ExistingComponent — 扩展 props: xxx
```

**⬡ 决策草稿（条件附加）**：若本次方案中存在以下任一情况，在方案末尾追加决策草稿块：
- 选择了有明显替代方案的技术路线
- 故意绕开通用模式（如不走统一错误处理中间件）
- 某模块引入了"不能随意改动"的约束
- 依赖了一个将来可能消失的外部限制

草稿格式：
```
⬡ 待记录决策：[topic]
决定：一句话描述
原因：关键驱动因素
禁止：NEVER ... 因为 ...
```

---

## Step 3：强制人工确认卡点

输出文档后，停止所有操作，显示以下提示：

```
───────────────────────────────────────
方案评审文档已生成。
在继续之前，请逐字段检查：
□ 需求理解：它说的是你想要的吗？
□ 实现思路：你能用自己的话复述吗？
□ 会动哪些东西：有没有它不该动的地方？
□ 完整例子：这个例子符合你的预期吗？
□ 没选的方案：它的理由你接受吗？
□ 不确定的地方：它承认的不确定你能接受吗？
□ 验收标准：这样验收够不够？
有任何一项不确定，回复你的问题，不要回复"继续"。
确认所有项目后，回复"确认，继续执行"。
───────────────────────────────────────
```

**在收到"确认，继续执行" / "同意" / "LGTM" / "可以开始"之前，不执行任何后续操作。**

---

## Step 4：决策兜底（仅 auto-dev 执行）

若方案中包含决策草稿，用户确认方案的同时视为确认决策草稿，**立即调用 `record-decision` skill 写入 `docs/decisions/`**，无需再次询问。
