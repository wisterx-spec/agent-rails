# Sync LLM Context

随着项目的持续开发，供给未来大模型 Agent 探索项目的上下文引导地图会产生信息盲区和滞后。
执行本 Skill，即授权 Agent 全局扫描工程的最新状态，提取**陷阱禁区、依赖入口和开发命令**，并对索引文件进行**增量修正**。

> 输出文件路径来自 `project.config.json → llm_context` 字段：
> - `core_doc`：最高优先级禁区声明，强制不超过 40 行
> - `backend_doc`：后端防查坑约束，强制不超过 30 行
> - `frontend_doc`：前端契约规则，强制不超过 30 行

---

## 🎯 产出规则 (Constraints)

大模型提示词文件绝不可写成冗长发散的长篇大论。更新时，必须遵循极其克制的紧凑注释格式（Key-Value 或简单短语列举）：
1. **core 文档**：只放最高优先级的禁区声明、防错挂载点和主干模块入口文件，**强制不超过 40 行**。
2. **backend 文档**：按需加载项，只放后端的防查坑约束、验证命令、高风险迁移操作必查点，**强制不超过 30 行**。
3. **frontend 文档**：按需加载项，只放前端契约规则、组件禁用约定、状态流防连带删除死角，**强制不超过 30 行**。

> ⚠️ 全局铁律：在任何情形下都**不要删除现有的成熟教训，仅进行追加或在已有行内修正**！
> 绝对禁止：修改任何项目源代码本身，更不可碰写 `.agents/rules/` 体系下的基础宪典。

---

## 🕵️ Phase 1: Scan & Discover (特征搜集)

Agent 启动本阶段后，需调用全局搜索/读取能力检索最新的目录结构树和关键词匹配：

- **后端领域**（`project.config.json → tech_stack.backend_path`）：
  重点挖掘通过装饰器动态注册的路由、隐藏的调度器，以及容易被静态分析误删的动态引用。

- **前端领域**（`project.config.json → tech_stack.frontend_path`）：
  排查全局状态管理中经过隐式调用的解构状态引用点。

- **基础设施**（`Makefile`, `package.json`, `pyproject.toml`, 根级 `README`）：
  寻找诸如启动服务、环境调试、快速压测的最频发运行指令。

对于文件，主要**判定并提取 4 大核心特征**：
1. **真实入口模块**：被频繁作为基类、入口等核心 import 依赖的基础文件
2. **高风险防重写禁区**：如并发共享锁、第三方重度交互协定
3. **静态分析避坑指南**：如由于依赖注入导致的未使用警告
4. **统一最佳实践**：特定的通用业务模式（如何抛出特定异常，如何规范化测试 Mock）

## ✍️ Phase 2: Condense & Push (生成增量更新)

完成挖掘后，将提炼的知识碎片按照对应领域分类，运用文件操作工具追加到指定的 LLM context 文档中。
- 如果遇到数量过于庞大的列表（例如 40 个路由），切记**将其抽象概括转化为 grep 检索指导命令**（如：*"修改前执行 `ls {{BACKEND_PATH}}/routers` 查验实时目录"*）。

---

## 🏁 Phase 3: Reporting (验证与汇报)

结束写库动作后，通过标准消息通知用户以下执行账单：
1. **更新摘要**：总结向这三个文件新增了哪些维度的条目和指令。
2. **需要复核的潜在地雷池 (`[UNVERIFIED]`)**：如果在扫描代码时，发现了一些可能缺乏测试防护/容易被后续脚手架误杀的隐秘写法，但无法十分笃定这是否算是"禁区"，将其记录在反馈列表内，供资深人类工程师最终拍板。
