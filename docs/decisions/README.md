# 架构决策记录 (Architecture Decision Records)

> 这里记录项目中**非显而易见的技术决策**：为什么这样设计、不能随意改动的原因。
>
> AI 在接触相关代码前会先读对应的决策记录，避免"善意地破坏"已有的设计。

---

## 使用规则

**什么时候写**：
- 做了一个"换个人来可能会改掉"的技术决定
- 选了非主流方案，原因不在代码里
- 某个模块有隐性约束，违反后会出问题但不会立刻报错
- 放弃了一个看起来更好的方案，原因值得记录

**什么时候不用写**：
- 行业通用最佳实践（不需要解释为什么用 HTTPS）
- 临时方案（标注在代码注释里即可，不需要在这里留存）
- 纯粹的偏好选择（没有约束意义）

**写完后**：在下方索引里加一行，供 AI 快速路由。

## 写法参考

好的决策记录回答四个问题：
1. **当时的处境**：面临什么约束或问题
2. **做了什么决定**：一句话
3. **为什么**：驱动因素，尤其是那些将来可能消失的约束
4. **禁止什么**：哪些"改进"会破坏这个决策
